# Release v1.3.0

- Date: `2026-03-12`
- Scope: performance, tenant consistency, queue hardening, admin operations
- Status: ready for commit after final validation

## Included

- `Dashboard` and `Logs` default to `7 days` with server-side expansion for larger periods.
- `Dashboard` stats now use a materialized view-backed RPC.
- `message_logs` queries were reduced to required columns and received supporting indexes.
- `enqueue-campaign` now sends minimal QStash payloads.
- Campaign completion moved to `complete_campaign_if_finished(...)`.
- Campaign counters and per-tenant circuit breaker were added in Postgres and wired into processing functions.
- Impersonation/tenant consistency was corrected across dashboard, logs, chats, and `Disparo`.
- Public/admin Edge Function access was split more safely:
- `auth_manager_dispara_lead` stays public only for recovery.
- Invite flows now require an authenticated user.
- `manage-users` now requires JWT validation.
- Sensitive frontend calls use authenticated helper requests instead of the public helper.

## Known behavior

- A campaign can end as `completed` even if one or more message logs finish as `failed`; completion currently means batch processing ended.
- The QA test used a synthetic destination, so the log remained `Falha` while the pipeline itself completed correctly.
